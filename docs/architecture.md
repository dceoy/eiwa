# Architecture

Eiwa is a static single-page app plus one minimal Cloudflare Worker route. It
has no database, no server-side LLM, and no user accounts.

## Overview

```text
Browser
  ├─ Preact SPA (src/App.tsx)
  │    ├─ dictionary lookup (src/dict.ts) ──> public/dict/**/*.json (static)
  │    └─ lookup orchestrator (src/lookup.ts)
  │           └─ optional AI engine (src/llm.ts)
  │                  └─ Web Worker (src/llm-worker.ts)
  │                         └─ @mlc-ai/web-llm, loaded from a CDN at runtime
  │                                └─ WebGPU (on-device model inference)
  └─ Service Worker (Workbox, generated by vite-plugin-pwa)
       └─ precaches the small app shell only

Cloudflare Workers Static Assets
  ├─ dist/**  (HTML/CSS/JS/icons/dictionary shards) served directly, no
  │           Worker invocation, when a request matches a file
  └─ worker/index.ts (Worker script)
        ├─ GET /api/health -> {"ok":true,"service":"eiwa"}
        └─ everything else -> env.ASSETS.fetch(request) (SPA fallback)
```

Almost all traffic is served as static assets with zero Worker invocation.
The Worker only runs for `/api/health` and for paths that don't match a
static file (which it then forwards to the asset handler, which applies the
SPA `not_found_handling` fallback).

## Request routing (Cloudflare Workers Static Assets)

`wrangler.jsonc` configures:

```jsonc
"assets": {
  "directory": "./dist",
  "not_found_handling": "single-page-application",
  "binding": "ASSETS"
}
```

`run_worker_first` is scoped to `["/api/health"]` only, so:

1. If the request path matches a file under `dist/`, serve it directly — the
   Worker is never invoked, and it doesn't count against Worker request
   limits.
2. `/api/health` always invokes the Worker's `fetch` handler first, which
   returns the health JSON directly. Without this, Cloudflare's
   `single-page-application` fallback intercepts browser navigation requests
   (`Sec-Fetch-Mode: navigate`) to unmatched paths — including `/api/health`
   — and serves `index.html` before the Worker ever runs, so a plain browser
   visit to `/api/health` would silently return the SPA shell instead of
   JSON. Non-navigation requests (e.g. `fetch()`/`curl`) were never affected,
   since only navigation requests get the SPA fallback treatment.
3. Every other unmatched path falls through to `env.ASSETS.fetch(request)`
   via the default asset handling, which applies
   `not_found_handling: "single-page-application"` and returns `index.html`
   for client-side routes.

This keeps the dynamic Worker surface to a single explicitly-routed route,
per the free-tier guardrails in issue #6.

## Dictionary shard flow

1. `fixtures/dict/{en,ja}.json` hold hand-authored fixture entries (see
   `src/dict-types.ts` for the `DictionaryEntry` shape).
2. `scripts/build-dict-shards.ts` (run by `pnpm run build` via `dict:build`)
   normalizes each headword (`src/dict-normalize.ts`), shards English by
   first ASCII letter and Japanese by first character, and writes:
   - `public/dict/<lang>/<key>.json` — `{ entries: Record<string, DictionaryEntry[]> }`
   - `public/dict/manifest.json` — schema version, build timestamp, shard
     list with `sha256:` checksums and entry counts, and deduplicated
     license/source metadata.
3. `scripts/validate-dict.ts` (`dict:validate`, also run by `pnpm run build`)
   re-reads the generated output, recomputes each shard's checksum, and
   validates every entry's shape (`src/dict-validate.ts`).
4. In the browser, `src/dict.ts` fetches the manifest once, then fetches and
   validates only the specific shard a lookup needs, caching both in
   CacheStorage (`src/dict-cache.ts`). A dictionary miss returns `[]`, not an
   error.

Production ingestion of real sources (JMdict/EDICT, optionally
Wiktextract) can later populate additional shards through the same
manifest/shard format — see `docs/release.md`.

## Lookup orchestration

`src/lookup.ts` is the single module aware of both the dictionary and the AI
engine; UI components only ever consume the `LookupEvent`s it yields:

```text
normalize input (src/normalize.ts)
  -> detect direction: en-ja | ja-en (src/language.ts), or use the user's
     Auto/EN→JA/JA→EN override
  -> look up dictionary entries (src/dict.ts)              [renders immediately]
  -> if an AI engine is already `ready`: generate a structured explanation
  -> merge dictionary facts + AI output (src/result-merge.ts) into an
     EiwaResult (src/result-schema.ts)
```

`lookup()` never triggers model loading itself. If no AI engine was passed,
or the engine isn't already `ready` (e.g. still downloading, or the user
never enabled it), the result is dictionary-only with `sourceKinds:
["dictionary"]`. This is what keeps model loading tied to an explicit user
action (the Settings → "Enable AI explanations" toggle) rather than any
particular lookup.

### Dictionary vs. AI separation

`src/result-merge.ts` builds the final `EiwaResult` so that
`dictionary`, `etymology`, `pronunciation`, and `derivedWords` are derived
**only** from dictionary entries — the AI's response schema
(`AiExplanationOutput` in `src/result-schema.ts`) doesn't even have fields for
those, so a malformed or hallucinated model response cannot overwrite or
invent a dictionary fact. The AI can only ever contribute `translation`,
`correction`, `nuance`, and `warnings`. `src/provenance.ts` labels each
rendered section `Dictionary`, `AI explanation`, or `Dictionary + AI`
accordingly.

## WebLLM engine

`src/llm.ts` is the facade the rest of the app calls
(`createLocalAiEngine()` → `ensureReady()` / `explain()` / `dispose()`); no
other module imports `@mlc-ai/web-llm` directly.

- **Lazy loading.** The model is never fetched until the user flips
  "Enable AI explanations" in Settings. `isWebGpuSupported()` gates this —
  if WebGPU isn't available, the toggle is disabled and the app stays
  dictionary-only.
- **Web Worker.** `CreateWebWorkerMLCEngine` runs the actual model in
  `src/llm-worker.ts` via `WebWorkerMLCEngineHandler`, keeping the UI thread
  responsive during load and generation.
- **Structured output.** `explain()` requests
  `response_format: { type: "json_object" }`, streams the response (so a
  live "generating…" state is possible), then parses and validates the
  accumulated JSON against `AiExplanationOutput` before it's ever rendered.
  Invalid JSON is treated as `generation-invalid-json`, not a crash.
- **Cancellation.** An `AbortSignal` maps to `engine.interruptGenerate()`.
- **Models.** `src/model-config.ts` offers a "Lite" (`Qwen2.5-0.5B-Instruct`,
  ~400 MB) and "Balanced" (`Qwen2.5-1.5B-Instruct`, ~1 GB) choice.

### Why WebLLM is loaded from a CDN

`@mlc-ai/web-llm` ships as a single pre-bundled ESM file
(`lib/index.js`, ~6 MB). Cloudflare Workers Static Assets rejects any
uploaded file over 5 MiB
(https://developers.cloudflare.com/workers/static-assets/billing-and-limitations/) —
this was discovered by actually running `wrangler deploy --temporary` after
wiring up the engine, not just by building locally. Bundling the library
into our own static assets is therefore not possible under this hosting
model.

Instead, `src/webllm-cdn.ts` dynamically imports the library from jsDelivr
(`https://cdn.jsdelivr.net/npm/@mlc-ai/web-llm@<version>/lib/index.js`) at
runtime, both from the main thread (`src/llm.ts`) and from inside the Web
Worker (`src/llm-worker.ts`, which buffers any messages that arrive before
the dynamic import resolves). The package has no bare-specifier imports of
its own, so a plain CDN file URL works without needing an ESM-resolving
proxy. `@mlc-ai/web-llm` stays a **devDependency** purely for its TypeScript
types (`import type` / `typeof import(...)`, fully erased at build time) —
it contributes nothing to the production bundle. If the CDN is unreachable,
`ensureReady()` rejects like any other model-load failure and the app stays
usable in dictionary-only mode.

Because dynamic `import()` has no `integrity` attribute, `loadWebLlm()`
verifies the library manually instead of trusting jsDelivr: it `fetch()`es
the file as bytes, hashes them with `crypto.subtle.digest("SHA-256", …)`,
and compares against `WEBLLM_SHA256`, a hash pinned next to
`WEBLLM_VERSION` in `src/webllm-cdn.ts`. Only on a match does it wrap the
bytes in a `Blob` and `import()` the resulting `blob:` URL — a mismatch
throws before any fetched code ever executes. Both the main thread
(`src/llm.ts`) and the Web Worker (`src/llm-worker.ts`) go through this same
function. `WEBLLM_SHA256` must be recomputed by hand whenever
`WEBLLM_VERSION` is bumped (instructions are in a comment next to it);
`src/webllm-cdn.test.ts` checks `WEBLLM_VERSION` against the
`@mlc-ai/web-llm` devDependency pin in `package.json` so the two can't drift
silently, though it can't verify the hash itself was recomputed correctly.

The model weights themselves (hundreds of MB to ~1 GB) are always fetched
by WebLLM directly from Hugging Face at runtime, cached by the browser —
they were never a static-asset candidate in the first place.

## PWA shell

`vite-plugin-pwa` (Workbox `generateSW` strategy) precaches only the app
shell — `index.html`, the main JS/CSS bundle, the manifest, and icons
(~74 KiB total). Dictionary shards (`dict/**`) are explicitly excluded from
the service-worker precache and from the SPA navigation fallback denylist,
since `src/dict-cache.ts` already caches them deliberately and on demand
under a separate CacheStorage bucket. `/api/*` is likewise excluded from the
navigation fallback so the Worker's real health route is never masked by a
cached `index.html`. `registerType: "autoUpdate"` means a new deployment is
picked up automatically rather than trapping users on stale assets
indefinitely.

## Error taxonomy

`src/errors.ts` defines the stable, user-facing error codes used throughout
the pipeline: `empty-input`, `unsupported-language`, `dictionary-miss`,
`webgpu-unavailable`, `model-load-failed`, `generation-cancelled`,
`generation-invalid-json`, `network-unavailable-for-dictionary-shard`. Every
failure mode degrades to a dictionary-only (or empty) result rather than
breaking the UI.
